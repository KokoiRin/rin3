## 我们究竟在建模什么

假设要做一个视频编辑器，用户可以在作品中添加贴纸。我们很容易写出一个 `Sticker` 类，给它加上开始时间、结束时间和图层，再用一个列表保存所有贴纸。

但有了这些类，并不等于有了领域模型。它们目前只是在记录数据，还没有表达这个业务世界怎样运转。

:::detail
“领域”是软件要解决的那一部分现实问题；“模型”则是我们为了理解和处理问题，对这部分现实做出的有意简化。地图不会复制整座城市，它只保留对当前目的有用的道路、站点和边界。领域模型也是如此：它不需要收纳全部业务资料，只需要准确表达那些会影响系统行为的概念、关系和规则。

因此，领域模型也不是数据库表的另一种画法。一张 `sticker` 表当然可以告诉我们贴纸有哪些字段，却不会自动告诉我们：一段怎样的时间范围才合法，两个看起来完全相同的贴纸是不是同一个贴纸，谁有权把第 31 个贴纸加入作品。

真正有用的领域模型会把这些答案放进代码。数据和行为待在一起，业务规则不再只是散落在页面、接口和 Service 中的几段 `if`。Martin Fowler 对 Domain Model 的简短定义正抓住了这一点：它同时包含行为与数据。

接下来我们不从术语表开始，而是跟着几条业务规则往下走。值对象、实体和聚合不会作为一套需要背诵的名词突然出现；每个概念都会用来解决上一步留下的问题。
:::
:::

:::slide {"kind":"code","chapter":"question","eyebrow":"START WITH RULES"}
## 从一条规则开始

产品对贴纸提出了四个要求：它不能出现在视频时间之外；一个作品最多有 30 个贴纸；同一图层在同一时刻不能重叠；复制出来的贴纸必须能够单独编辑和删除。

```kotlin
data class Sticker(
    val id: String,
    var startMs: Long,
    var endMs: Long,
    var layer: Int,
)

class StickerProject {
    val stickers = mutableListOf<Sticker>()
}
```

这段代码能保存贴纸，却无法保证任何一条规则。

:::detail
任何调用者都可以把 `startMs` 改成负数，把 `endMs` 改到 `startMs` 前面，或者绕过添加流程直接向列表塞入第 31 个贴纸。系统虽然拥有名为 Sticker 的对象，却仍要依赖每个调用者“自觉地做对”。

这时首先需要理解的不是聚合，而是**不变量**。不变量是模型在任何对外可见的时刻都必须满足的业务条件。`0 <= start < end` 是一条不变量，“一个作品最多有 30 个贴纸”也是一条不变量。一次表单校验只检查某个入口，而不变量要求无论操作来自页面、API、批处理还是测试，模型都不能进入非法状态。

不变量听起来像是在增加限制，实际效果却是减少系统必须处理的可能性。如果开始和结束时间可以分别随意修改，代码就不得不面对“开始时间已经更新，结束时间还没更新”的半成品状态。禁止这种状态之后，模型失去了一些自由，其他代码却得到了一条可靠前提：拿到的时间范围一定可用。

所以，建模的第一个动作不是挑选某种 DDD 构件，而是问：**哪些状态在业务上根本不应该存在？** 找到这些规则，才知道模型需要保护什么。
:::
:::

:::slide {"kind":"code","chapter":"derive","eyebrow":"VALUE AND IDENTITY"}
## 值与身份

时间范围是否合法，只取决于开始和结束两个值。我们可以把它们合成一个不可分割的 `StickerTimeRange`，在创建时就排除非法组合。

```kotlin
data class StickerTimeRange private constructor(
    val startMs: Long,
    val endMs: Long,
) {
    companion object {
        fun create(startMs: Long, endMs: Long): StickerTimeRange {
            require(startMs >= 0)
            require(endMs > startMs)
            return StickerTimeRange(startMs, endMs)
        }
    }
}
```

这就是一个**值对象**：它描述“是什么值”，相等性由内容决定，通常也保持不可变。

:::detail
两个 `StickerTimeRange(0, 3000)` 没有必要分清谁是谁。只要起止时间相同，它们表达的业务含义就相同。把经常一起出现的原始类型收进值对象，还有另一个好处：规则有了唯一的家。其他代码不必在每次使用时间范围时重新检查起止顺序。

不过，值对象只能保护自己掌握的信息。它可以保证 `end > start`，却不知道当前视频究竟有多长。“不能超过视频结尾”需要同时看到时间范围和作品时长，不能偷偷塞进一个并不了解作品的值对象里。

输入的含义也会影响边界处理。用户拖动贴纸时越过视频结尾，如果产品规定“拖过头等于拖到结尾”，交互层可以先吸附到合法位置，再创建时间范围；加载一份已经保存的工程文件时，同样的越界更可能意味着文件损坏，此时静默裁剪反而会掩盖问题。值对象负责保证进入模型的值合法，但不需要规定所有非法输入都以同一种方式处理。

贴纸本身则不同。两个贴纸即使文字、位置和时间范围完全相同，也必须能够分别移动和删除。我们关心的不只是“这些属性是什么”，还关心“这是哪一个贴纸”。这种拥有稳定身份、会经历状态变化的对象叫作**实体**。

这也解释了为什么复制贴纸必须生成新的 `StickerId`。复制属性，是为了得到一个外观相同的新实体；如果连 ID 一起复制，我们得到的就不是两个独立对象，而是同一个身份的两份冲突表示。相反，移动或改字不会改变 ID，因为变化前后仍是原来那个贴纸。

值对象和实体的区别不是“小对象与大对象”，而是业务如何判断它们相同。时间范围按内容相等；贴纸按身份延续。
:::

## 一致性边界

单个贴纸可以保护自己的时间范围，却不知道作品里已有多少贴纸，也看不见其他贴纸占用了哪些图层。跨越多个贴纸的规则，需要一个能够看到整体状态的对象来裁决。

在这个例子里，`StickerProject` 就承担这个角色。它和边界内的 Sticker 共同组成一个**聚合**，而 `StickerProject` 是外部修改这个整体时必须经过的**聚合根**。

> 聚合不是一棵“对象关系树”，而是一条业务一致性边界。

:::detail
成为聚合根以后，`StickerProject` 不能只提供一个公开列表。它需要收回关键状态的修改权，通过 `addSticker()`、`moveSticker()`、`removeSticker()` 这类带有业务含义的方法改变内部对象。添加之前，它检查数量和图层；移动之前，它重新检查目标位置。检查与修改发生在同一个入口，规则才不容易被绕过。

这并不意味着与贴纸有关的数据都要放进聚合。素材作者、下载地址和运营标签当然与编辑过程有关，但它们并不参与“最多 30 个”和“图层不能重叠”的即时判断。把它们一起加载和保存，只会让边界更重。

判断聚合边界时，应该问的不是“这些对象有没有关系”，而是“哪些状态必须在一次业务操作完成时共同保持合法”。只要业务允许短暂不一致、明确的中间状态或失败后的补偿，就未必需要放进同一个聚合。聚合越大，加载成本和并发冲突通常也越大，因此应在守住强一致性规则的前提下尽量保持小。

聚合还带来一个容易忽略的问题：它只能根据自己看到的状态作判断。假设两个请求同时读到一个已有 29 个贴纸的作品，它们各自在内存中添加一个贴纸时都合法，但先后保存后却会得到 31 个。常见做法是给聚合记录版本号：第一个请求保存后版本变化，第二个请求用旧版本提交时失败，然后重新加载并再次由聚合裁决。

因此，聚合与并发控制解决的是相邻的两个问题：聚合判断“按照我现在看到的状态，这次变化是否合法”；版本控制保证“提交时，我看到的状态仍然没有过期”。幂等键只能防止同一个请求被重复执行，不能代替这种并发检查。
:::

## 模型之外

到这里，领域模型已经能判断一次修改是否符合业务规则。但一次真实操作还需要读取数据库、查询视频信息、开启事务和返回结果。这些工作很重要，却不应该全部塞进聚合。

领域模型不是整个应用程序。它需要边界外的代码把一次用例组织起来。

:::detail
例如“添加贴纸”可以由一个 `AddStickerUseCase` 负责：接收用户意图，从 Repository 加载最新的 `StickerProject`，取得需要的外部事实，构造合法的时间范围和新 ID，调用 `project.addSticker()`，最后保存聚合。

这类对象通常叫**应用服务**。它掌握的是流程顺序，而不是业务裁决。它知道先加载、再调用、最后保存，却不应该自己写出“如果已有 30 个就拒绝”这种规则。否则同一个模型从另一个入口使用时，规则又会开始复制。

**Repository** 则让应用服务以聚合为单位加载和保存对象。它隐藏数据库和映射细节，使领域模型不必认识 SQL、ORM 或远程接口。Repository 的接口通常围绕聚合根设计，这不是命名习惯，而是因为聚合正是我们选择的一致性与保存边界。

有些业务计算确实不自然地属于某个实体或值对象。例如允许移动的区域同时取决于贴纸类型、画布和安全区，把它硬塞进 Sticker 或 Project 都显得牵强。这时可以使用**领域服务**表达这条业务能力。领域服务属于领域，因为它计算的是业务规则；它又不拥有业务状态，最好保持显式输入、显式输出。

领域服务和应用服务名字相似，判断方式却很直接：前者回答“业务上怎么算”，后者回答“一次用例先做什么、后做什么”。纯函数则是另一条维度：普通加法可以是纯函数却不是领域服务，一个不访问外部状态的安全区计算可以既是领域服务又是纯函数。
:::
:::

## 一次完整的操作

1. 页面提交 `AddStickerCommand`，描述用户想做什么。
2. 应用服务加载最新聚合和必要的外部事实。
3. 值对象先保证输入在局部上合法。
4. 聚合根据整体状态做最终裁决并修改自己。
5. Repository 带版本号保存聚合。

这条链路最重要的不是步骤数量，而是谁拥有最后决定权。

:::detail
页面可以提前判断按钮是否可用，也可以在拖动时显示越界提示。这些预判能改善体验，却不是业务规则的权威入口。页面持有的数据可能已经过期，同一个命令也可能来自 API、快捷键或批处理。真正执行命令时，聚合仍然必须重新检查。

同样，视频分析服务可以告诉我们“这段视频长 12 秒”，安全区算法可以返回“贴纸允许出现在这个矩形内”。这些都是聚合做决定所需的事实。外部负责取得事实，领域服务可以负责计算事实，聚合则根据当前状态和这些事实决定是否接受变化。

这条分工可以概括为一句话：**外部提供事实，聚合做出裁决。** 如果外部服务直接返回 `canAdd = true`，聚合只是照做，那么规则仍然掌握在模型之外；如果聚合自己去访问网络查询所有信息，它又会与基础设施绑在一起。把事实作为输入传入，能同时守住业务权威和技术边界。
:::
:::

:::slide {"kind":"prose","chapter":"collaborate","eyebrow":"DOMAIN EVENT"}
## 事件不是通知

贴纸添加成功后，界面要刷新预览，编辑器要支持撤销，审计系统还可能要永久记录是谁添加了贴纸。这些后续动作看起来都像“通知”，却不应该都叫领域事件。

**领域事件**表达的是一个已经发生、并且值得进入业务语言的事实。

:::detail
如果业务确实关心“贴纸已添加”这件事，而且聚合之外有独立的响应者，那么模型可以产生 `StickerAdded`。它只描述事实本身，例如作品 ID、贴纸 ID、操作者和发生时间。谁订阅它、是否使用消息队列、怎样重试和可靠投递，则是模型之外的架构问题。

界面刷新通常只是当前状态变化后的显示反应，用普通状态观察就够了。撤销与重做关心的是用户执行过哪些编辑命令，更适合保存在 Command History 中。审计记录关心的是不可抵赖的业务事实，才更可能消费领域事件。撤销历史和审计历史有时包含相似字段，但前者服务于交互回退，后者服务于追责与留痕，不能因为都叫“历史”就合成一个机制。

判断是否需要领域事件时，可以问三个问题：业务人员会明确说出这件事吗？聚合之外是否有人独立关心它？聚合是否不应该认识这些响应者？只有答案把我们推向解耦时，事件才有价值。每改一个字段就发布一个事件，只会把内部实现细节伪装成领域语言。
:::
:::

## 回到这张图

现在再看这些术语，它们不再是一排孤立的定义，而是在回答一组前后相连的问题。

| 当我们追问…… | 出现的概念 | 它提供的答案 |
| --- | --- | --- |
| 哪些状态绝不能出现？ | 不变量 | 定义模型必须持续满足的业务条件 |
| 这只是一个值，还是某个具体对象？ | 值对象、实体 | 分别用内容与身份表达业务概念 |
| 谁能看见并保护跨对象规则？ | 聚合、聚合根 | 划出一致性边界并控制修改入口 |
| 规则不属于任何单个对象，怎么办？ | 领域服务 | 表达无自然对象归属的业务计算 |
| 一次操作怎样接入数据库和外部系统？ | 应用服务、Repository | 编排用例，并按聚合加载和保存 |
| 模型产生的重要事实怎样越过边界？ | 领域事件 | 用业务语言描述已经发生的事实 |

:::detail
如果只记住一条建模路径，可以记住下面这条：先找出不允许出现的业务状态，再沿着规则所需的信息寻找最小而完整的所有者。只依赖一组值的规则，可以由值对象保护；需要跨越生命周期辨认同一个对象，就需要实体；跨越多个内部对象的强一致性规则，则把我们推向聚合。

只有当模型本身已经清楚，才讨论应用服务怎样编排、Repository 怎样保存、领域服务怎样提供计算、领域事件怎样把事实传向边界之外。否则这些“外围构件”很容易反过来吞掉业务逻辑，留下只有字段和 getter/setter 的贫血模型。

这也解释了领域模型的真正目标。它不是尽可能多地使用 DDD 术语，也不是为每张表建立一个类，而是让代码拥有与业务相同的语言和约束：非法状态难以构造，状态变化必须经过有意义的行为，而每条重要规则都有一个真正能够负责的地方。
:::
:::
